>本文记录的是我从产生搭博客的想法到实际搭完的操作过程，如果你对我从产生想法到解决的心路感兴趣，你可以往下阅读。但如果你想从本文学习如何从零搭建一个博客，那要先声明，本文的操作过程更适合有编程基础的人跟着操作。毫无编程基础的读者可能只能获得一些关键词，需要自行额外学习。

搭建个人博客的想法在我心中盘踞已久，应该有年许有余了，但我本人拖延症严重，想到有一堆问题要我去解决就懒得搞，是要自己用主流框架搞一个项目？还是找个博客搭建框架？搞好之后又如何部署？域名咋搞？终于，抱着学习的心态，花了大概几个小时，把它们都搞定了，本文记录一下解决问题的思考和过程，当然，如果你恰好想搭一个且有接触过编程，看完本文照着做应该就能搭一个。
### 一、自己用前端框架撸一个还是找个博客搭建框架？
1. 自己撸
   * 自己撸有什么好处？
     * 可以找点不常用的框架玩，像我目前工作主要用vue2、react，那我可以趁着这个机会搞个vue3项目玩玩
     * 不需要受框架束缚，主题也好、功能也好，利用我已有的技术栈都能随心的开发，像这个，就是我特别喜欢的样例 [翁天信的个人网站](https://www.dandyweng.com/)

   * 自己撸有什么缺点？
     * 网页的设计咋整？photeshop零基础，审美又挫，那我大概率找一个喜欢的现成组件库来开发，那我的博客最后不是一股浓浓的后台管理风？
     * 主题的开发虽然自由了，但是博客内容要咋整？每篇文章都去开发一个组件？那不可能，谁也没有这个功夫的。那自己研究一下怎么在Vue里面渲染markdown？或者找一个把markdown解析成html的js库，再想办法跟Vue项目结合起来，比如说，直接嵌入dom节点？虽然这些思路似乎都是可行的，但想到这里我已经觉得很复杂了，开发量肯定不小，头大。
2. 找博客搭建框架
   * 找博客搭建框架有什么好处？
     * 快！几行命令，几分钟，如果没有主题定制需求，你的博客就已经搭建好了！
     * 即便需要主题，也可以去主题商店找其它人搭建好的，接着又是几行命令，几个配置，博客主题也就焕然一新了，香！
   * 找博客搭建框架有什么缺点？
     * 主题用人家的，页面的样式或者功能可能不能完全满足自己的需求，如果不想动脑子开发的话，几乎是人家提供什么你就只能用什么，当然，我后面发现其实对于博客这件事来说博客搭建框架提供的功能大部门都够用了，样式也是可以基于他人的主题付出一定的学习成本后（远低于自己撸的时间成本）去修改的，我这属于瞎操心。

写到这里我已经觉得自己很傻了，典型的想得太多，很明显找个框架搭起来这件事就做完了，非给自己设想问题阻挠自己。那选什么呢？作为一名前端，选择了基于node.js的hexo框架。

### 二、用Hexo搭建博客网站的步骤
对于前端工程师来说，hexo的使用几乎没有学习成本。因此我这里只是列一下命令步骤并稍加解释，如果期间还有不懂的，建议去读官方文档 [hexo](https://hexo.io/zh-cn/docs/)。
1. 安装node（跟装QQ微信没什么区别，下载安装包一路安装即可）
2. 安装hexo脚手架工具, 打开任意命令行工具 执行:  
```
  npm i -g hexo-cli
```
3. 用脚手架工具建立项目
```
  hexo init xx    //  xx是你的建立的项目的文件夹名称
```
4. 进入项目文件夹安装依赖
```
  cd xx  // xx同上
  npm install
```
5. 到这一步你的博客项目就搭建完成了，你可以把它托管到一个git仓库，这样你的博客就上云了，可以多地多端同步和开发。这个初始博客项目具有清晰的项目结构，关于内部每个文件夹的作用，还是建议读官方文档  [项目结构](https://hexo.io/zh-cn/docs/setup)。

但哪怕你读不懂或者不去读这个文档也是可以的，你只要记住，每次你想写一篇新文章，就执行：
```
  hexo new xx // xx为你的文章名称
```
而你的所有文章都会存储在项目结构的_posts文件夹内，打开该文件夹内的对应的文章，则可以用markdown开始写你的文章内容啦。
```
├── _config.yml
├── package.json
├── scaffolds
├── source
|   ├── _drafts
|   └── _posts  // markdown文章都在这里
└── themes
```
在写的过程中，如果想即时预览写的效果，可以在命令行执行：
```
  hexo server // 启动一个静态服务器
```
执行后，会在本地的4000端口自动一个http服务器，在浏览器输入`locallhost：4000`则可以即时的看到自己的网站的最新的样子。<br>
写完了之后，你的内容都是用markdown写的，而我们向读者提供的，其实是一个个网页，即本质是一堆html文件，因此，我们需要hexo帮我们完成这一步的转换。通过执行：
```
  hexo generate
```
执行完后，你的项目中会多出一个`public`文件夹
```
├── _config.yml
├── package.json
├── scaffolds
├── public  // 生成好的html文件都在这里
├── source
|   ├── _drafts
|   └── _posts  // markdown文章都在这里
└── themes
```

这个public文件夹的内容，就是你的博客网站的内容啦。

### 三、怎么部署我的博客网站供他人访问

因为我们的网站都是静态资源，只需要一个静态文件服务器就可以了，但自己买个云服务器，去上面启动一个http服务器等等操作都太复杂了，实际上，github对于静态服务器的需求给我们提供了很好的解决方案----**github pages**。

github pages是github提供的静态服务器解决方案，使用的方法非常简单。

1. 建立一个github仓库，仓库命名为`你的github用户名.github.io`，例如，我的github用户名为liantianyun，则我的仓库名为`liantianyun.github.io`
2. 接着，在这个仓库内放置一个`index.html`文件，完成！

接着，你就可以直接在浏览器输入liantianyun.github.io直接访问这个html文件，就是这么简单！github免费为我们提供了一个静态服务器，不需要我们花一分钱，也不用到服务器内操作任何东西！

接着，只要把public文件内的内容放到这个仓库的根目录就可以了。

因此，有人是这样配置自己的解决方案的，一个仓库专门用来写内容，一个仓库则是刚刚提到的这个静态服务器仓库，每次`hexo generate`后的public文件夹内容都手动提交到这个仓库，就这样：写博客+部署都有了各自的解决方案。

但这不是最优的方案，实际上，没有必要拆出两个仓库来，我们可以用两个分支来解决这个问题。
